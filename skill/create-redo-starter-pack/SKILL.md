# Create Redo Starter Pack

Stand up a new merchant's core marketing automations in one session, each
email brand-customized by Claude rather than copied from a generic
template.

Composes with:
- `create-redo-email` — template generation (prompts, schemas, copy-craft,
  brand archetypes, anti-slop). Every email in the pack uses it.
- `create-redo-welcome-automation` — the welcome series is exactly that
  skill's output; reuse it wholesale.
- `stop-slop` — prose review.

## What's in the pack

| Flow | Trigger (`key` / `schemaType`) | Emails | Cadence | Default? |
| --- | --- | --- | --- | --- |
| Welcome series | `email_signup` / `email_marketing_signup` | 3 | Day 0 / 2 / 5 | ✅ on |
| Abandoned cart | `cart_abandoned` / `marketing_cart_abandonment` | 2 | 1h / 24h | ✅ on |
| Browse abandonment | `browse_abandoned` / `marketing_browse_abandonment` | 1 | 4h | ✅ on |
| Checkout abandonment | `checkout_abandoned` / `marketing_checkout_abandonment` | 2 | 1h / 24h | ⬜ opt-in |

**Default pack = welcome + cart + browse** (6 emails, 3 flows). The
universal high-ROI three.

**Checkout abandonment is opt-in** because it overlaps with cart for many
merchants (someone who started checkout also has items in cart). If the
merchant wants both, mention Redo's skip-condition fields
(`isCartAbandoned` / `isCheckoutAbandoned`) so the two flows don't
double-message — though wiring skip conditions is a v2 enhancement, not
in this skill yet.

**Not in v1** (different schemas / transactional / data-bound blocks —
deserve their own pass):
- Order confirmation (transactional, `order_tracking` schema, uses
  order-summary blocks, different recipient field names)
- Post-purchase review request (`reviews` schema, `customerName` not
  `customerFullName`)
- Win-back / re-engagement (segment- or date-triggered, more config)
- Text concierge (exists already as `createConciergeAutomationIfMissing`
  on the merchant surface — Redo seeds it at onboarding)

## Inputs to collect

1. **Brand description** — what they sell, voice, hero imagery, real
   product names. Feeds the archetype router + every email's content.
2. **Session JWT** — read from `~/.redo/jwt`, where the user stores it once;
   never ask them to paste it into the chat (see INSTALL.md). Needs
   `MANAGE_TEMPLATES` + `MANAGE_CAMPAIGNS` + `MANAGE_AUTOMATIONS`.
3. **Which flows** — confirm the default (welcome + cart + browse), offer
   checkout abandonment as an add-on.
4. **Discount** — for the welcome series (default `WELCOME10`, 10%, 14-day
   expiry). Abandonment flows usually run without a discount on a starter
   pack (don't train customers to abandon for a coupon); offer it as an
   option but default to no-discount on cart/browse/checkout.
5. **Signup source** — `email_signup` (Redo forms) vs
   `email_signup_shopify` (Shopify consent), for the welcome trigger.

## Pre-flight checks (BEFORE any API call)

**Load the token from the user's local store — never ask the user to paste the
JWT into the chat, and never echo it** (no `echo "$TOKEN"`, no `curl -v`). It
must stay out of the conversation. If `~/.redo/jwt` is missing, stop and tell
the user to run `connect.sh` from the claude-marketing-email repo (details in
INSTALL.md); do not accept a pasted token.

```bash
TOKEN="$(cat ~/.redo/jwt)"   # Keychain alt: security find-generic-password -s redo-jwt -w
```

Shell state doesn't persist between commands, so re-set `TOKEN` at the start of
every block that calls the API. Decode and check the payload — prints only the
claims, never the token:

```bash
cut -d. -f2 ~/.redo/jwt | base64 -d 2>/dev/null | jq '{aud, exp, sub}'
```

- `aud` = `mcht/<teamId>` → capture `teamId`.
- `sub` → capture `userId` for `newFlow.createdByUserId`.
- `exp` in the future.
- Shopify connection (`storeUrl` ~ `*.myshopify.com`) — only matters if a
  flow uses a discount. No Shopify → skip discount creation, warn.

## Workflow

1. **Collect + confirm inputs.** Show the resolved pack composition (which
   flows, cadences, discount yes/no per flow, signup source). Wait for
   confirmation.
2. **Pre-flight checks.** Decode JWT, capture `teamId` + `userId`. Shopify
   check if any flow uses a discount.
3. **Pick brand archetype** once, using
   `create-redo-email/references/best-practices/_index.md`. The whole pack
   shares one archetype + brand voice. Confirm with the user.
4. **Create the discount** if the welcome series (or any flow) uses one —
   `createDiscount`, capture `_id`. Skip if no Shopify.
5. **Generate all templates.** For each flow, generate its email(s) using
   `create-redo-email`'s generation prompt + the archetype + copy-craft.
   Each email's role differs (see
   [references/pack-composition.md](references/pack-composition.md) for the
   per-flow content briefs). Apply `team`, `templateType: "default"`,
   `category: "Marketing"`, `schemaType: "marketing_email"` to every
   template.
6. **Show the user a preview** of the whole pack — every email's
   section-by-section summary, subject lines, cadence, discount placement.
   This is a lot of content; group it by flow. Wait for "go".
7. **POST every template** via `createEmailTemplate` (NOT
   `createSavedEmailTemplate` — see the two-templates trap). Capture each
   `output._id`, keyed by flow + position.
8. **Create every flow** via `createAdvancedFlow` (`/rpc/` namespace), each
   wired to its templates. Use the exact step shapes in
   [references/pack-composition.md](references/pack-composition.md). Ship
   each `enabled: true`.
9. **Return builder URLs** for every flow:
   `https://app.getredo.com/stores/<teamId>/marketing/email-sms/automations/<automationId>`.
   Summarize: flows created, total emails, discount code, any placeholder
   images to replace, and a "send yourself a test from each flow" nudge.

## Idempotency

Before creating each flow, the merchant may already have one with the same
name (e.g. they ran this before, or Redo seeded defaults). Mirror the
`createConciergeAutomationIfMissing` pattern: optionally list existing
automations first and skip / rename collisions. At minimum, name flows
distinctly (e.g. include the brand or a date suffix) so a re-run doesn't
silently create duplicates. Ask the user how to handle a collision rather
than blindly duplicating.

## The hour-wait trap (critical for abandonment flows)

The welcome series uses **day** waits, where `numDays` alone is enough.
Abandonment flows use **hour** waits, and the scheduler computes delay via
`waitStepSeconds(step)`:

```ts
if (step.numSeconds !== undefined) return step.numSeconds;
else if (step.numDays !== undefined) return step.numDays * SECONDS_IN_A_DAY;
```

So **a wait without `numSeconds` is interpreted in DAYS.** For any
hour-or-minute wait you MUST set `numSeconds` explicitly, or a "1 hour"
wait silently becomes 1 day. Example 1-hour wait:

```json
{ "type": "wait", "numDays": 1, "numSeconds": 3600, "timeUnit": "Hours", "nextId": "..." }
```

`numDays` is the displayed count in the chosen unit (poorly named — it's
really "time value"), `timeUnit` is the display unit, `numSeconds` is what
actually schedules the send. Set all three consistently. Full conversions
in [references/pack-composition.md](references/pack-composition.md).

## Status / confidence

- **Welcome series** — live-tested end to end (via
  `create-redo-welcome-automation`). High confidence.
- **Abandonment flows** — schema-derived, **NOT yet live-tested.** The
  hour-wait `numSeconds` shape and the abandonment trigger field names
  (`customerEmail` / `customerFullName`, shared across all marketing
  triggers)
  are derived from source, not a successful live POST. Test against a
  Shopify-connected store before treating as production-ready. Same
  open-question caveat as the welcome flow's first run.

## Errors

Inherits every error mode from `create-redo-email` (template POST) and
`create-redo-welcome-automation` (flow POST). The pack-specific ones:

| Symptom | Cause | Fix |
| --- | --- | --- |
| Abandonment "1 hour" wait fires after 1 day | `numSeconds` omitted on the wait step | Set `numSeconds` explicitly; `numDays` alone = days |
| HTTP 400 Zod on abandonment trigger | Wrong `key`/`schemaType` pair | cart: `cart_abandoned`/`marketing_cart_abandonment`; browse: `browse_abandoned`/`marketing_browse_abandonment`; checkout: `checkout_abandoned`/`marketing_checkout_abandonment` |
| Abandonment email "no recipient" at send | Wrong recipient field name | All marketing triggers use `customerEmail` / `customerFullName` (NOT `email`/`firstName` — the local seed script uses those but they're untested) |
| Duplicate flows after re-run | No idempotency guard | List existing automations first; skip or rename by name |
| Partial pack (some flows created, then a 500) | The opaque `createEmailTemplate` 500 on malformed-storeUrl sandboxes | The flows already created are fine; resume from the failed template. Retest on a real Shopify sandbox. |
